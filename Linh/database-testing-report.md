# Database Testing Report for EShop

Date: 2026-07-04

## 1. What This Report Is About

This report explains how I tested the database behavior of the EShop project.

The EShop project uses:

- Backend: Node.js + Express
- Database: SQLite
- Frontend: React

The required database testing tools were:

- DbUnit
- Database Rider
- Tonic.ai Fabricate

DbUnit and Database Rider are normally used in Java projects. Because this
EShop project is not Java, I created a separate Java testing folder:

```text
database-testing/
```

This folder does not change the EShop app. It only tests the EShop app.

## 2. Scenario Chosen

The test scenario is:

```text
Admin imports products from a CSV file.
```

This scenario is useful because the EShop requirements say:

- CSV product import must validate each row.
- Product name must not be empty.
- Product price must be positive.
- If one CSV row is invalid, the whole import should rollback.
- Admin APIs should only be usable by admin users.

## 3. What Correct Behavior Should Look Like

If the app worked correctly:

1. A CSV row with an empty product name should be rejected.
2. A product with a negative price should be rejected.
3. If one row in the CSV is invalid, no rows from that CSV should be inserted.
4. A normal user should not be able to use the admin import API.

## 4. What Actually Happened

The test found 3 real problems.

### Problem 1: Import Does Not Roll Back

The CSV contained one invalid row with an empty product name.

Expected:

```text
No products should be inserted.
```

Actual:

```text
Some products were still inserted.
```

API response:

```text
Import hoàn tất: 2/3 sản phẩm được thêm
```

This means the app inserted 2 products even though the CSV had an invalid row.

### Problem 2: Negative Price Is Saved

The test sent a product with a negative price.

Expected:

```text
The product should be rejected.
```

Actual:

```text
The product was saved in the database.
```

Example from the database:

```text
Product: TONIC-FABRICATE-Desk Lamp
Price: -12.5
```

This is wrong because product prices should be positive.

### Problem 3: Normal User Can Use Admin Import API

The test logged in as a normal user and called:

```text
/api/admin/import-products
```

Expected:

```text
The request should be rejected.
```

Actual:

```text
The normal user successfully imported a product.
```

API response:

```text
Import hoàn tất: 1/1 sản phẩm được thêm
```

This is a security problem because admin APIs should check that the user is
really an admin.

## 6. JavaScript Scripts Used

There are two JavaScript files in the `database-testing/scripts/` folder.

They are not part of the EShop app. They are helper scripts for testing.
They are mainly used for metrics. The actual bug-finding assertions are in the
Java Maven test `EshopCsvImportScenarioTest`.

### `run-eshop-csv-import-scenario.mjs`

This script uses normal hand-written test data.

What it does:

1. Logs in as the admin user.
2. Creates and logs in as a normal user.
3. Sends bad CSV-style product data to the EShop import API.
4. Sends one import request as a normal user.
5. Copies the SQLite database from the Docker backend container.
6. Checks what products were actually saved.
7. Writes timing and result data into `database-testing/artifacts/`.

In simple words:

```text
This script measures the app with normal test data.
```

### `run-tonic-fabricate-ai-scenario.mjs`

This script uses the CSV generated by Tonic Fabricate.

It reads this file:

```text
database-testing/tonic-ai/ai-variant-products.csv
```

Then it performs the same test steps as the first script.

In simple words:

```text
This script measures the app with AI-generated test data.
```

## 7. Commands Used

### Run DbUnit and Database Rider hello-world tests

```bash
cd database-testing
mvn test -q
```

Result:

```text
Passed
```

Important: this command only proves the tools are installed and their
hello-world tests work. The EShop scenario is skipped here because it needs the
running Docker app.

### Run the EShop database scenario

```bash
cd database-testing
mvn test -q -Deshop.e2e=true \
  -Deshop.baseUrl=http://localhost:3000 \
  -Deshop.backendContainer=eshop-sut-backend-1 \
  -Dtest=EshopCsvImportScenarioTest
```

Result:

```text
Failed, because the database tools found bugs in EShop.
```

This is the important bug-finding command. The test asserts the correct
requirements. Since the current EShop implementation is wrong, Maven reports
test failures.

The failing checks are:

```text
csvImportShouldRollbackWhenAnyRowIsInvalid
csvImportShouldRejectNegativePrice
csvImportShouldRejectNonAdminUser
tonicFabricateCsvShouldFollowSameRules
```

Example Maven failure messages:

```text
FR-16 failed: CSV import must be all-or-nothing. expected: <0> but was: <1>
FR-16 failed: product price must be positive. expected: <0> but was: <1>
FR-12 failed: normal user should not access admin import API. Response was 200
```

The Tonic Fabricate AI data is included in the Java Maven test through
`tonicFabricateCsvShouldFollowSameRules`.

### Run the JavaScript metrics helper for the Tonic Fabricate AI version

```bash
cd database-testing
node scripts/run-tonic-fabricate-ai-scenario.mjs
```

This script records timings and stores a JSON artifact. It is not the main proof
that the bugs exist; the Maven test failures above are the proof.

## 10. Final Summary

The database testing work is complete.

DbUnit and Database Rider were installed and passed their hello-world tests.

The EShop CSV import scenario was tested against the real running app on
`localhost`.

Tonic Fabricate generated AI test data, and that data was used to run the same
scenario again.

The testing found 3 real bugs:

1. CSV import does not rollback when one row is invalid.
2. Negative product prices can be saved.
3. Normal users can access an admin import API.
